# Архитектура solver_bench

## Обзор

`solver_bench` — инструмент для бенчмаркинга прямых разреженных солверов (MUMPS, SuperLU, Intel MKL PARDISO) на матрицах формата Matrix Market (`.mtx`). Для каждой комбинации (матрица × солвер × число потоков) собираются время факторизации, время решения, относительная невязка и пиковое потребление памяти. Результат записывается в CSV.

---

## Архитектура: оркестратор / воркер

Бинарный файл работает в двух режимах, определяемых наличием флага `--worker` в аргументах.

```
solver_bench (оркестратор)
│
├── читает матрицы, оценивает число обусловленности
│
└── для каждой (матрица × солвер × потоки):
      fork() → /usr/bin/time -v → solver_bench --worker ...
                                         │
                                         ├── загружает матрицу
                                         ├── генерирует правую часть
                                         ├── запускает солвер
                                         └── stdout: key=value строки
```

**Зачем subprocess?**

- Каждый запуск получает чистый OMP-пул — нет влияния предыдущих запусков.
- При зависании или OOM процесс убивается через `SIGKILL` без падения оркестратора.
- Пиковая RSS снимается через `/usr/bin/time -v` со стороны (парсится из stderr).

---

## Формат хранения матриц (CSC)

Все матрицы хранятся в формате **Compressed Sparse Column (CSC)**, 0-индексированный.

```
col_ptr[j]   .. col_ptr[j+1]-1  — диапазон ненулей в столбце j
row_idx[k]                       — строка k-го ненуля
values[k]                        — значение k-го ненуля
```

При чтении симметричных `.mtx` нижний треугольник достраивается автоматически — матрица хранится полностью. Каждый солвер при необходимости конвертирует CSC в свой формат.

## Модули

### `matrix_io` — чтение Matrix Market

Читает `.mtx` файл:
1. Проверяет заголовок `%%MatrixMarket`.
2. Пропускает строки с `%`.
3. Парсит строку размерности `M N NNZ`.
4. Читает триплеты `(row, col, val)`, конвертирует в 0-индекс.
5. Для симметричных матриц добавляет зеркальные элементы (кроме диагонали).
6. Сортирует по столбцу, строит CSC (`col_ptr`, `row_idx`, `values`).

Бросает `std::runtime_error` на любой ошибке парсинга.

---

### `rhs_gen` — генерация правой части

```
x_true ← uniform(-1, 1)  (фиксированный seed, по умолчанию 42)
b = A * x_true
```

Решение `x_true` известно заранее — это позволяет проверить точность солвера через относительную невязку.

---

### `cond_est` — оценка числа обусловленности

Используется точный метод через LAPACK:

1. **`dlange`** — вычисляет 1-норму матрицы `‖A‖₁`.
2. **`dgetrf`** — LU-факторизация (in-place, плотная).
3. **`dgecon`** — оценивает `rcond = 1 / κ(A)` через обратную итерацию.
4. Возвращает `κ = 1 / rcond`.

Ограничение: плотная LU требует O(n²) памяти. При n > 10 000 (≈ 800 МБ) функция возвращает `-1.0`, что записывается как пустое поле в CSV. Для несквадратных матриц также возвращает `-1.0`.

---

### `metrics` — относительная невязка

```
rel_residual = ‖A·x − b‖₂ / ‖b‖₂
```

Вычисляется в воркере после получения решения `x` от солвера. Для нулевого вектора `b` возвращает `-1.0` (пустое поле).

---

### Солверы

Все солверы реализуют интерфейс `SolverBase::solve(A, b, x_true)` и возвращают `SolveResult` с полями `status`, `time_factorize_sec`, `time_solve_sec`, `rel_residual`.

#### MUMPS (`mumps_solver`)

Прямой мультифронтальный солвер. Формат входных данных — **COO 1-индексированный** (отдельные массивы строк `irn` и столбцов `jcn`). Конвертируется из CSC при запуске.

Фазы MUMPS:
- `job = -1` — инициализация
- `job = 4` — анализ + факторизация (замеряется)
- `job = 3` — решение (замеряется)
- `job = -2` — освобождение памяти

MUMPS требует MPI даже в однопроцессном режиме. MPI инициализируется однократно перед первым вызовом.

Настройки для плохо обусловленных матриц: `icntl[7]=77` (скалирование строк и столбцов), `icntl[12]=1` (улучшенный выбор опорного элемента).

#### SuperLU (`superlu_solver`)

Солвер на основе суперузловой LU-факторизации. Принимает CSC 0-индексированный напрямую (копия, т.к. `dgssv` модифицирует массивы).

Вызов `dgssv` выполняет анализ, факторизацию и решение за один шаг — поэтому `time_solve_sec = 0`, всё время записывается в `time_factorize_sec`.

После любого вызова `dgssv` структуры `L`, `U` освобождаются безусловно — SuperLU инициализирует указатели store до факторизации.

#### PARDISO (`pardiso_solver`)

Intel MKL PARDISO. Требует **CSR 1-индексированный с отсортированными столбцами в каждой строке**. Конвертация из CSC выполняется через подсчёт вхождений в строки, перестановку элементов и сортировку каждой строки.

Фазы PARDISO:
- `phase = 12` — анализ + факторизация (замеряется)
- `phase = 33` — решение (замеряется)
- `phase = -1` — освобождение внутренней памяти (выполняется после записи результата, ошибка cleanup не перезаписывает статус OK)

Настройки для плохо обусловленных матриц: `iparm[10]=1` (скалирование), `iparm[12]=1` (matching-based reordering), `iparm[20]=1` (Bunch-Kaufman пивотинг).

---

### `orchestrator` — внешний цикл

Для каждой комбинации (матрица × солвер × потоки):

1. Читает матрицу, оценивает `κ` один раз для всех солверов.
2. Вызывает `launch_worker` — делает `fork()`, в дочернем процессе запускает `exec("/usr/bin/time -v solver_bench --worker ...")`.
3. Два `std::thread` читают stdout и stderr дочернего процесса в строки.
4. Если задан таймаут — опрашивает `waitpid` каждые 200 мс; по истечении шлёт `SIGKILL`.
5. После завершения процесса (или kill) **оба треда join'ятся** — только затем строки парсятся.
6. RSS извлекается из stderr `time -v` (строка `Maximum resident set size (kbytes):`), переводится в МБ через `round(kb / 1024.0)`.
7. Строка пишется в CSV немедленно, без буферизации.

---

### `worker` — внутренний режим

1. Устанавливает `OMP_NUM_THREADS` через `setenv` — солверные библиотеки подхватят при первой инициализации OMP-пула.
2. Читает матрицу, генерирует `b = A * x_true`.
3. Создаёт экземпляр солвера через `make_solver` (зависит от скомпилированных флагов `HAVE_MUMPS`, `HAVE_SUPERLU`, `HAVE_PARDISO`).
4. Вызывает `solver->solve(A, b, x_true)`.
5. Пишет результат в stdout в формате `key=value\n`.

---

### `cli` — разбор аргументов

| Флаг | Описание |
|------|----------|
| `--matrices <path>` | Директория с `.mtx` или один файл |
| `--solvers <list>` | Через запятую: `mumps,superlu,pardiso` |
| `--threads <list>` | Числа потоков: `1,2,4,8` |
| `--output <path>` | CSV-файл (по умолчанию `results/bench.csv`) |
| `--timeout <sec>` | Таймаут на запуск (по умолчанию 300, 0 = без таймаута) |
| `--seed <uint>` | Seed для генерации `x_true` (по умолчанию 42) |

Флаги воркер-режима (`--worker`, `--matrix`, `--solver`) используются оркестратором внутри, напрямую не вызываются.

---

### `csv_writer` — запись результатов

Пишет CSV-строки немедленно (`flush` после каждой). Поля со значением `-1` (или `< 0`) записываются как пустые — в таблице будет пустая ячейка.

Колонки:

| Поле | Описание |
|------|----------|
| `matrix_name` | Имя файла без расширения |
| `rows`, `nnz`, `density` | Размер и заполненность матрицы |
| `cond_est` | Число обусловленности (пусто для n > 10000) |
| `solver` | `mumps` / `superlu` / `pardiso` |
| `status` | `OK` / `FAIL` / `TIMEOUT` |
| `time_factorize_sec` | Анализ + факторизация |
| `time_solve_sec` | Только решение (0 для SuperLU) |
| `rel_residual` | `‖Ax − b‖₂ / ‖b‖₂` |
| `memory_mb` | Пиковая RSS воркер-процесса |
| `iterations` | Всегда пусто (прямые солверы) |
| `threads` | `OMP_NUM_THREADS` для этого запуска |

## Поток данных

```
.mtx файл
    │
    ▼
read_mtx → SparseMatrix (CSC, 0-indexed)
    │
    ├── estimate_cond → κ (LAPACK dgecon, n ≤ 10000)
    │
    └── [fork] run_worker
              │
              ├── generate_rhs → x_true, b
              │
              ├── solver->solve(A, b, x_true)
              │       │
              │       ├── MUMPS:   CSC → COO 1-indexed → dmumps_c
              │       ├── SuperLU: CSC 0-indexed → dgssv
              │       └── PARDISO: CSC → CSR 1-indexed → pardiso
              │
              └── stdout: status=OK\ntime_factorize=...\n...
                              │
                    [orchestrator парсит]
                              │
                    CsvWriter → results/bench.csv
```
